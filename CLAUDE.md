# プロジェクト: 生命保険 アウトバウンドコール 営業成績分析

## データパイプライン

```
python generate_data.py   # data/calls_raw.csv（ノイズあり）を生成
python clean_data.py      # data/calls.csv（クレンジング済み）を生成
pytest tests/ -v          # 31ケース（test_raw_* + test_clean_*）を実行
streamlit run app.py      # ダッシュボード起動
```

## 設計原則

### ノイズ設計
- ノイズは必ず「現実の原因」を付与する（コメントに記載）
- 処理方針は問題の性質で使い分ける:
  - 除外: レコード自体が信頼できない場合（call_hour範囲外）
  - 中央値補完: サンプルバイアスを避けたい連続値欠損
  - '不明'補完: 分析側で制御できるよう明示したいカテゴリ欠損
  - フラグ付与: 除外ではなく可視化で判断させたい場合（重複コール）

### テスト構造
- `test_raw_*`: ノイズが意図通り注入されているかを検証
- `test_clean_*`: クレンジング後のビジネスルールを保証

### app.py セクション構成
1. KPIサマリー
2. 担当者別パフォーマンス
3. 統計的有意性検定（Spearman / カイ二乗 / Kruskal-Wallis）
4. 時間帯別分析
5. 顧客属性クロス分析
6. 商品別成約実績
7. 月次トレンド
8. 成約予測モデル（Random Forest + 5-Fold CV AUC）

### SQLの品質基準
- Window関数（RANK/LAG/SUM OVER/NTILE）を必ず含める
- 多段CTEでファネル分析・シミュレーションを実装
- BigQuery固有構文（PERCENTILE_CONT）も別途記載

## メモ
- サンプル数（5,000件）の根拠: 15名×月20コール×12ヶ月≈3,600件が現実的最低ラインで
  統計的検出力確保のため5,000件に設定
- ノイズ件数・割合は generate_data.py のコメントテーブルに記載済み
